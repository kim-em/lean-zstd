# Track C2a: convert `decodeZeroRepeats` to well-founded recursion

Issue: #5
Branch: `agent/da859622`

## Summary

Replaced the `fuel : Nat` parameter in `decodeZeroRepeats` with a
well-founded recursion keyed on remaining bits in the `BitReader`.

## Changes

### `Zstd/Native/Fse.lean`

- Removed the `fuel : Nat` parameter from `decodeZeroRepeats`.
- Added `termination_by br.data.size * 8 - (br.pos * 8 + br.bitOff)`.
- Added a runtime guard `hadv` in the recursive branch so
  `decreasing_by exact hadv` closes trivially. When a successful
  `readBits 2` does not strictly advance the bit position (impossible
  on a well-formed reader), the function returns an error rather than
  looping.
- Updated the single caller in `decodeFseLoop` (dropped the `1000`
  fuel argument).
- Wrapped the def in `set_option linter.unusedVariables false in` to
  suppress the false-positive warning on `hadv` (the linter does not
  see references from `decreasing_by`).

### `Zstd/Spec/Fse.lean`

Updated the four dependent theorems away from `induction fuel`:

- `decodeZeroRepeats_cellCount`
- `decodeZeroRepeats_probs_ge_neg1`
- `decodeZeroRepeats_bitPos_ge`
- `decodeZeroRepeats_pos_le_size`

Each now uses the auto-generated `decodeZeroRepeats.induct` principle
via a `suffices aux : ∀ br probs sym …` wrapper. The four cases of the
induct principle map naturally to: (1) `readBits` errors, (2)
recursive call (uses the IH), (3) recursive call rejected by the
`hadv` guard (contradiction — the result is `.error`), (4) non-repeat
terminal branch.

The `probs_ge_neg1` case uses
`congrArg Prod.fst hpush'` to transport `pushZeros_probs_ge_neg1`
across the `hpush' : pushZeros … = (probs₁, sym₁)` equation at the
goal level, sidestepping motive-type issues that occur when rewriting
in a hypothesis with dependent `Fin` indices.
